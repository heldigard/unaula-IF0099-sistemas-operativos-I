Estructura de un Disco (HDD)

Componentes principales

Componentes Mecánicos

  • Platos: superficies magnéticas que giran (5400-15000 RPM)
  • Pistas: circunferencias concéntricas en cada plato
  • Sectores: porciones de una pista (unidad mínima, típico 512B-4KB)
  • Cabezal: lee y escribe datos (uno por superficie)
  • Cilindro: conjunto de pistas alineadas en todos los platos

Capacidades Típicas

  • Disco de 1TB: ~2 billones de sectores
  • Densidad: 100-1000 Gb/in²
  • Latencia HDD: 2-15 ms
  • Latencia SSD: 0.1-0.5 ms

Organización Lógica

Sector → Bloque → Cluster

El SO gestiona bloques, no sectores

Direccionamiento Físico

Dirección de disco = Cilindro + Cabeza + Sector

Ejemplo: (1024, 3, 7) = Cilindro 1024, Cabezal 3, Sector 7

CHS vs LBA

Esquema Nombre Descripción
CHS Cilindro-Cabezal-Sector Antiguo, usado por BIOS
LBA Logical Block Addressing Moderno, direccionamiento lineal

LBA simplifica la gestión

  • El SO ve bloques: 0, 1, 2, 3, ...
  • El controlador traduce a CHS automáticamente

Tiempo de Acceso

Fórmula del Tiempo Total

Tiempo total = seek + latencia rotacional + transferencia

Componente Descripción Tiempo típico
Seek Mover el cabezal a la pista correcta 5-10 ms
Latencia Esperar que el sector gire hasta el cabezal 2-8 ms
Transferencia Leer o escribir datos 0.1-1 ms

Ejemplo de cálculo:

8 ms (seek) + 4 ms (latencia) + 1 ms (transfer) = 13 ms

Nota: El seek suele ser el componente más lento, representando ~60% del tiempo total.

Algoritmos de Planificación de Discos

¿Por qué importa?

El problema: Las solicitudes de E/S llegan desordenadas
  • Si atendemos en orden de llegada (FCFS), el cabezal se mueve mucho
  • Mucho movimiento = mayor tiempo de espera = bajo rendimiento
Solución: Reordenar solicitudes para minimizar movimiento del cabezal

Algoritmos Principales

FCFS (First-Come, First-Served)

First-Come, First-Served

  • Atiende en orden de llegada
  • Simple pero ineficiente
  • Puede causar "oscilación" del cabezal

SSTF (Shortest Seek Time First)

Shortest Seek Time First

  • Atiende la solicitud más cercana
  • Eficiente pero causa inanición
  • Solicitudes lejanas esperan mucho

SCAN (Elevator)

Algoritmo del ascensor

  • Atiende en una dirección y luego regresa
  • Justo, pero espera variable
  • Similar a un ascensor

C-SCAN (Circular SCAN)

  • Atiende en una dirección
  • Regresa sin servir solicitudes
  • Tiempo de espera uniforme

LOOK / C-LOOK

  • Como SCAN/C-SCAN
  • Pero sin ir al extremo innecesario

Ejemplo Visual: Algoritmo SCAN

Solicitudes: 98, 183, 37, 122, 14, 124, 65, 67

Cabezal inicia en: 53

Movimiento (SCAN):

53 → 65 → 67 → 98 → 122 → 124 → 183 → 199
(regresa) → 37 → 14

Ventaja: tiempo de espera más uniforme que FCFS y SSTF

Tabla Comparativa de Algoritmos

Algoritmo Ventaja Desventaja Uso recomendado
FCFS Simple Alto movimiento Sistemas simples
SSTF Eficiente Inanición posible Carga baja
SCAN Justo Espera variable Uso general
C-SCAN Uniforme Regreso sin servicio Servidores
LOOK Eficiente Más complejo Alta carga
C-LOOK Muy eficiente Más complejo Producción

RAID (Redundant Array of Independent Disks)

¿Por qué RAID?

Problema: Los discos fallan y son lentos

Solución: RAID combina discos para:

  • Mejorar rendimiento (RAID 0)
  • Añadir redundancia (RAID 1, 5, 6, 10)
  • Ambos (RAID 10)

Niveles RAID Comunes

RAID 0 - Striping

  • Datos divididos entre discos
  • 2x velocidad (teórico)
  • Sin redundancia
  • Si falla 1 disco, se pierde todo

RAID 1 - Mirroring

  • Datos duplicados en 2+ discos
  • Lectura 2x más rápida
  • 50% de capacidad útil
  • Sobrevive 1 fallo

RAID 5 - Paridad Distribuida

  • Datos + paridad distribuida
  • Mínimo 3 discos
  • Sobrevive 1 fallo
  • Escritura más lenta

RAID 6 - Doble Paridad

  • Como RAID 5 pero 2 paridades
  • Mínimo 4 discos
  • Sobrevive 2 fallos simultáneos
  • Escritura aún más lenta

RAID 10 - Stripe of Mirrors

  • RAID 0 + RAID 1
  • Mínimo 4 discos
  • Rendimiento y redundancia
  • 50% capacidad útil

RAID en la Práctica

Uso RAID recomendado Por qué
Gaming PC RAID 0 o ninguno Máxima velocidad
Servidor web RAID 10 Rendimiento + redundancia
NAS hogar RAID 5 Balance costo/capacidad
Data center RAID 6 o 10 Alta disponibilidad
⚠️ Consideración importante: RAID NO es backup. Protege contra fallos de disco, no contra eliminación accidental, corrupción de datos, desastres, etc.

HDD vs SSD

Característica HDD SSD
Latencia 5-15 ms 0.1-0.2 ms
IOPS 100-200 10k-100k
Ruido Audible Silencioso
Resistencia Sensible a golpes Más resistente
Costo/GB Bajo Mayor
Regla práctica: SSD para sistema y aplicaciones, HDD para almacenamiento masivo.

NVMe vs SATA SSD

Diferencias clave

Aspecto SATA SSD NVMe SSD
Interfaz SATA III (6 Gb/s) PCIe 3.0/4.0 (32+ Gb/s)
Colas 1 cola, 32 comandos 65k colas, 64k comandos
Latencia ~50 µs ~10 µs
IOPS 100k 1M+
Ancho banda 560 MB/s 3500-7000 MB/s

¿Por qué NVMe es más rápido?

Arquitectura

  • Comunicación directa con CPU vía PCIe
  • Menos capas de protocolo
  • Comandos paralelos (múltiples colas)
  • Menor overhead

Ventajas Prácticas

  • Boot en segundos
  • Apps abren instantáneamente
  • Mejor para workloads pesados

Para Servidores

  • Más I/O por segundo
  • Menor latencia = DB más rápidas
  • Menor CPU utilizada para I/O

Consideraciones

  • Costo mayor que SATA
  • Requiere slot M.2 o PCIe
  • Compatible casi todas las motherboards modernas

Optimización de Rendimiento

Para HDD

  • Evitar desorden de archivos (defrag)
  • Reducir movimientos del cabezal (SCAN/LOOK)
  • Colocar archivos frecuentes juntos

Para SSD

  • Habilitar TRIM
  • Evitar desfragmentación
  • Dejar espacio libre (over-provisioning)
  • No llenar completamente
